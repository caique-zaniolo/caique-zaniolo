Architecture & System Design
The operating brain of Gravitas Energy — how every component connects

GravOS is not a chatbot. It is not a dashboard. It is a structured system of AI agents, context documents, and integration protocols that allow Gravitas to operate at institutional scale with a minimal human workforce. Every agent is a defined, repeatable worker. Every document it produces becomes institutional memory. Every email, meeting, and calendar entry becomes operational intelligence.

Design Principles

01
Signal over Noise
Every output must contain a decision-ready insight, not raw data
02
Predictable Outputs
Every agent produces the same structure every time it runs
03
Minimum Headcount
Design every workflow to run with the fewest humans possible
04
Compounding Memory
Every output feeds the Document Library for future agent use
05
Comms Awareness
Agents always have context from emails, calendar, and meetings
06
Bilingual by Default
All outputs available in English and Brazilian Portuguese
07
Proprietary First
Build internal IASPs before paying for third-party tools
08
Audit Trail Always
Every agent action is logged with source, date, and output reference

Agent Framework
How every Gravitas agent is built, run, and evaluated

A GravOS agent is a precisely defined AI worker. It has a fixed job, fixed inputs, fixed outputs, and fixed behavior rules. The goal is that any agent, run at any time by any user, produces an output that is structurally identical, contextually current, auditable, and actionable. Agents do not improvise structure. They do not add unsolicited opinions. They do not skip steps.

Agent Behavior Rules

These eight rules apply to every agent in every module without exception.

  • 01
    Source EverythingEvery factual claim must cite its source. Unverifiable claims are marked [UNVERIFIED] — never presented as fact.
  • 02
    Never Hide UncertaintyConfidence levels (High / Medium / Low) must be stated. A low-confidence output is more valuable than a falsely confident one.
  • 03
    Output Structure Is SacredSections cannot be added, removed, or renamed without updating the agent Identity Card. Every reader knows exactly where to find information.
  • 04
    Actions, Not ObservationsEvery output ends with a clear action. Not "PLD is elevated" but "PLD is elevated: consider locking in fixed-price supply for Q3 before the next PMO on [date]."
  • 05
    Comms Context Is Always CheckedBefore generating any output, agents check the Communications Layer for relevant emails, calendar events, and meeting notes from the past 14 days.
  • 06
    Bilingual ProtocolFollow 04_LANGUAGE_PROTOCOL for all language decisions. Default: English for internal strategy; Portuguese for counterparty-facing content.
  • 07
    Library FirstBefore generating new analysis, agents check the Document Library for existing relevant documents. Agents do not duplicate work.
  • 08
    Escalation Is Not OptionalIf an escalation condition is met, the agent escalates. It does not soften the alert or bury it at the bottom of an output.

Document 02
Document Library
Institutional memory — structure, taxonomy, and update rules

The Document Library is GravOS's institutional memory. It is a structured collection of living documents that agents read from and write to. Every analysis produced, every regulatory summary, every counterparty profile, every meeting summary — all of it lives here and is reused. Without the Library, agents have no memory. With it, every agent benefits from everything every previous agent has learned.

Key Rule

Agents never start from zero. They always check the Library first.

Library Architecture

/GRAVITAS_LIBRARY/ /LIB_MARKET/ PLD data, forward curves, ENA, hydrology /LIB_REGULATORY/ ANEEL, CCEE, ONS, MME publications and summaries /LIB_COUNTERPARTIES/ One file per counterparty - credit, history, risk /LIB_CONTRACTS/ One file per deal - terms, status, risk flags /LIB_PORTFOLIO/ Current positions, MtM, VaR, exposure /LIB_MEETINGS/ Meeting notes, decisions, action items /LIB_COMMS/ Processed email threads and summaries /LIB_INTELLIGENCE/ Competitive intelligence, sector news, macro /LIB_SYSTEM/ Agent QC logs, system health, version history

Counterparty File Standard — /LIB_COUNTERPARTIES/

Naming convention: CP_[CNPJ_LAST8]_[COMPANY_SHORT_NAME].md

Each counterparty file contains: Identity & CCEE Registration, Credit Score and recommended credit limit, Financial Summary (3-year history), KYC/Compliance Status, Deal History, Open Positions and Exposure, Communication Log, and active Risk Flags. This is the single source of truth for everything Gravitas knows about a company.

Contract File Standard — /LIB_CONTRACTS/

Naming convention: DEAL_[DEAL_ID]_[CP_SHORT_NAME]_[YYYYMM].md

Each contract file contains: Parties, all extracted commercial terms (volume, bandwidth, sazonalização, modulação, GSF clause, price, tenor), risk flags by severity, ETRM reconciliation status, settlement status, invoicing status, and all document references.

Meeting File Standard — /LIB_MEETINGS/

Naming convention: MTG_[YYYYMMDD]_[MEETING_TYPE]_[PARTICIPANTS_SHORT].md

Each meeting file contains: participant list, context summary, all decisions made with owners and deadlines, all open action items, market or deal intelligence captured verbally, and list of agents triggered by the meeting's content.

Communications Layer
Gmail, Google Calendar, and meeting transcript integration protocol

The Communications Layer is what separates GravOS from a generic AI tool. It ensures that every agent runs with live awareness of what is actually happening at Gravitas. An agent that knows the PLD forecast but not that a counterparty called yesterday to dispute an invoice is half-blind. The Comms Layer closes that gap.

ChannelToolWhat GravOS Extracts
EmailGmailDeal threads, counterparty commitments, regulatory notifications, invoice disputes, payment confirmations
CalendarGoogle CalendarScheduled meetings, deadlines, payment due dates, regulatory publication dates, CCEE settlement windows
MeetingsTranscripts / NotesDecisions made, commitments given, market intelligence shared verbally, action items with owners

Pre-Run Communications Check

When any GravOS agent runs, it follows this three-step check before generating output:

Step 1 — Email Scan (Last 14 Days)

The agent scans Gmail using structured keyword groups:

TopicKeywords (Portuguese / English)
PLD & MarketPLD, spot, "preco liquidacao", despacho, dispatch
Deal-relatedcontrato, proposta, oferta, termsheet, MWh, MWm
Counterparty[counterparty name], [CNPJ], [contact name]
Settlementliquidacao, CCEE, fatura, NF-e, pagamento, invoice
RegulatoryANEEL, resolucao, normativa, ONS, circular
Riskinadimplencia, default, disputa, atraso, overdue

Step 2 — Calendar Check (Today + 14 Days)

The agent scans Google Calendar for meetings with counterparties, internal deal reviews, regulatory deadlines (CCEE settlement close, ANEEL consultation periods), payment due dates on active contracts, and PMO/IPDO publication dates.

Step 3 — Meeting Notes (Last 14 Days)

The agent checks /LIB_MEETINGS/ for recent meeting summaries. It specifically looks for decisions that affect current analysis, commitments to counterparties not yet formalized in contracts, internal strategy changes that override previous assumptions, and market intelligence shared verbally that is not yet in /LIB_MARKET/.

The Morning Briefing Agent — 03_COMMS_MORNING_BRIEFING

Most Important Daily Output

The Morning Briefing runs daily at 07:30 BRT and gives Gravitas leadership and traders everything they need to start the day in one document. It combines market data, communications intelligence, calendar, portfolio alerts, collections watch, and regulatory watch.

The Morning Briefing output includes these sections: Urgent Items (act today, max 5 items), Market Pulse (PLD by sub-market with weekly trend, ENA vs. MLT signal), Communications Summary (last 18 hours of key email intelligence), Today's Agenda (calendar events with preparation notes), Portfolio Alerts (imbalances, VaR breaches, credit limit warnings), Collections Watch (invoices overdue or due within 5 days), and Regulatory Watch (expected ANEEL/CCEE/ONS publications).

Meeting Processor Agent — 03_COMMS_MEETING_PROCESSOR

When a meeting transcript, audio summary, or set of meeting notes is provided, this agent extracts all participants and roles, all decisions made ("agreed," "decided," "confirmed"), all commitments made to external parties, all action items with owners and deadlines, commercial intelligence (deals, pricing, market views), and any risk or compliance items. It saves the structured summary to /LIB_MEETINGS/ and identifies which agents should be triggered by the meeting's content.

Language Protocol
Bilingual operating rules — English and Brazilian Portuguese

Gravitas operates in Brazil. Its regulatory environment, counterparties, legal documents, and market data are overwhelmingly in Brazilian Portuguese. Its strategic architecture, investor communications, and international partnerships operate in English. GravOS agents must move fluently between both. The wrong language in the wrong document is not a cosmetic error. A contract summary in English for a Brazilian counterparty, or a regulatory alert that loses nuance in translation, is an operational risk.

Language Decision Tree

Is the output counterparty-facing? YES → Use Brazilian Portuguese (PT-BR) Exception: counterparty communicates in English → match their language NO → Is it regulatory or legal in nature? YES → PT-BR source summaries + English summary section NO → Is it internal strategy or investor-facing? YES → English NO → Is it a daily operational document? YES → Both languages (bilingual format) NO → Default: English

21_RM_CONTRACT_EXTRACTOR
Contract Extraction Agent
On Demand PT-BR + EN
Reads any energy contract (PDF, Word, email attachment) and extracts every commercially and legally relevant term into the standard Deal File format, flagging non-standard language and risk items automatically.
Process
1.  Identify document type: new contract / amendment / addendum
2.  Extract parties → verify CNPJ against counterparty library
3.  Extract all commercial terms:
    → Volume, bandwidth, sazonalizacao profile
    → Price (fixed / indexed — identify index)
    → Tenor: start and end date
    → Sub-market, product type (convencional / incentivada)
    → Modulacao rights if any
4.  Extract GSF clause → classify: Buyer / Seller / Shared / Absent
5.  Extract payment terms, penalty clauses, termination rights
6.  Compare each term against Gravitas standard template
7.  Flag deviations from standard template
8.  Identify embedded options: flexibility rights, early termination
9.  Identify missing fields → flag as [MISSING]
10. Assess overall contract risk: Low / Medium / High / Critical
11. Generate Deal File
12. Propose ETRM entry fields for human input

31_OC_SETTLEMENT_CHECK
Settlement Verification Agent
Monthly PT-BR + EN
Verifies that every active contract was settled correctly by CCEE — comparing metered volumes to contracted volumes, identifying descasamento, and reconciling settlement values against internal ETRM records.
Process
1. Load all contracts active in the settlement month
2. For each contract:
   a. Extract contracted volume (MWm) → apply sazonalizacao profile
   b. Load CCEE metered volume for the counterparty
   c. Calculate descasamento (contracted minus metered)
   d. Apply PLD → calculate financial value of descasamento
   e. Compare to ETRM settlement record
   f. Flag any variance between CCEE and ETRM
3. Aggregate total settlement position for Gravitas
4. Classify each contract: Settled correctly / Variance / Dispute required
5. Calculate net settlement receivable or payable
6. Trigger INVOICE_GEN for correctly settled contracts
7. Trigger DISPUTE_ANALYZER for disputed contracts
Escalation Rules
  • Critical Total settlement variance exceeds R$50,000 → escalate to Operations + Risk Officer
  • Critical A counterparty's metered volume is zero (potential default / non-delivery)
  • Critical ETRM vs. CCEE mismatch on more than 20% of contracts
  • Medium Single contract descasamento exceeds 15% of contracted volume
31_OC_INVOICE_GEN
Invoice Generation Agent
Triggered PT-BR Only
Generates fully validated NF-e data for each settled contract with correct Brazilian tax treatment, cross-referenced against contract terms. Ensures zero invoice errors before issuance.
Tax Validation Checklist
Tax ItemCheck
ICMSVerify origin/destination state pair — apply correct interstate rate
PIS / COFINSApply correct rate based on counterparty's tax regime (lucro real / presumido)
DIFALCalculate if applicable (interstate B2C or special regime)
ISSConfirm not applicable for energy supply (verify)
Five-Point Validation Gate
Volume confirms with CCEE settlement     [PASS / FAIL]
Price confirms with contract terms        [PASS / FAIL]
Counterparty CNPJ verified               [PASS / FAIL]
Tax treatment correctly applied           [PASS / FAIL]
Payment terms match contract              [PASS / FAIL]
————————————————————————————————
RESULT: APPROVED for NF-e issuance / BLOCKED — [reason]
Note: Invoice must NOT be issued if any check fails.
Escalation Rules
  • Critical Any validation check fails → invoice blocked, escalate to Operations Lead
  • Critical Counterparty credit limit would be exceeded by this invoice
  • Medium Invoice value differs more than 5% from ETRM expected settlement
  • Medium Tax treatment is ambiguous → flag for tax advisor review
31_OC_COLLECTIONS_MANAGER
Collections Strategy Agent
Weekly EN + PT-BR
Manages the entire collections process — tracking all outstanding receivables, predicting payment behavior, and recommending the right strategy for each counterparty at the right time.
Aging Classification
StatusDaysRecommended Action
MonitorDue in more than 10 daysAutomated tracking only
Prepare contactDue in 1–10 daysPrepare payment reminder
First contactOverdue 1–5 daysPolite email reminder (PT-BR draft generated)
Escalated contactOverdue 6–15 daysPhone call + formal email, credit score refresh
Legal reviewOverdue more than 15 daysEscalate to Director, consider legal action

For each overdue invoice, the agent generates a PT-BR communication draft addressed to the correct contact at the counterparty, with tone calibrated to the relationship risk profile and days outstanding.

Escalation Rules
  • Critical Invoice overdue more than 15 days with no communication response → legal review
  • Critical Total overdue exposure exceeds counterparty's approved credit limit
  • Critical Pattern: 3 consecutive late payments from same counterparty → credit score review
  • Medium Invoice overdue more than 5 days with no email response → phone contact

Handling Escalations

Never Ignore a Critical Escalation

If an agent outputs an escalation with Critical or High severity, review and act on it before running any further agent outputs on the affected deal or counterparty. The escalation format always states: Issue (one sentence), Risk Level, Recommended Action, and the Deadline by which a human must act.
